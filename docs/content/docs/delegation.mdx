description: Delegating permissions from one agent to another with depth limits and cascading revocation.

## What are delegation chains

A delegation chain lets one agent grant a subset of its permissions to another agent. This pattern is common in agentic systems where an orchestrator spins up sub-agents to complete specific tasks and wants each sub-agent to have only the access it needs.

Key rules:
- The delegated permissions must be a subset of the parent agent's permissions.
- Chains have a maximum depth — how many hops a delegation can travel.
- Chains have an expiry date after which they are no longer valid.
- Revoking a chain is immediate and cascades to any chains that depended on it.

## Creating a delegation

```typescript
const chain = await kavach.delegate({
  fromAgent: orchestrator.id,   // must be active and have the permissions
  toAgent: subAgent.id,         // receives the permissions
  permissions: [
    { resource: 'mcp:github:issues', actions: ['read'] },
  ],
  expiresAt: new Date(Date.now() + 3_600_000),  // 1 hour
  maxDepth: 2,
});

console.log(chain.id);          // dlg_...
console.log(chain.depth);       // 1
console.log(chain.expiresAt);   // Date
```

The `fromAgent` must currently hold every permission being delegated. If it does not, the call throws with `INSUFFICIENT_PERMISSIONS`.

## Permission subset enforcement

Suppose the orchestrator has:

```typescript
{ resource: 'mcp:github:*', actions: ['read', 'write', 'comment'] }
```

Valid delegations (subsets):

```typescript
{ resource: 'mcp:github:issues', actions: ['read'] }           // ✓ narrower resource, subset of actions
{ resource: 'mcp:github:*', actions: ['read'] }                // ✓ same resource, fewer actions
{ resource: 'mcp:github:repos', actions: ['read', 'comment'] } // ✓ narrower resource, same actions
```

Invalid delegations:

```typescript
{ resource: 'mcp:github:*', actions: ['delete'] }     // ✗ action not held by parent
{ resource: 'mcp:slack:*', actions: ['read'] }        // ✗ resource not held by parent
```

## Depth limiting

`maxDepth` controls how many additional hops the delegation chain can be extended. With `maxDepth: 2`, the sub-agent can delegate to a sub-sub-agent, but that sub-sub-agent cannot delegate further.

```typescript
// Orchestrator → Sub (depth 1, maxDepth 2)
await kavach.delegate({
  fromAgent: orchestrator.id,
  toAgent: sub.id,
  permissions: [{ resource: 'mcp:github:issues', actions: ['read'] }],
  expiresAt: new Date(Date.now() + 3_600_000),
  maxDepth: 2,
});

// Sub → SubSub (depth 2, maxDepth 2) — allowed
await kavach.delegate({
  fromAgent: sub.id,
  toAgent: subSub.id,
  permissions: [{ resource: 'mcp:github:issues', actions: ['read'] }],
  expiresAt: new Date(Date.now() + 1_800_000),
  maxDepth: 1,
});

// SubSub → anything — blocked, depth limit reached
```

The default `maxDepth` when not specified is 3.

## Cascading revocation

Revoking a chain invalidates it and all chains that were created downstream of it:

```typescript
await kavach.delegation.revoke(chain.id);
```

If orchestrator → sub → subSub is a chain, revoking orchestrator → sub also revokes sub → subSub immediately. Sub-agents that relied on those permissions will get `allowed: false` on their next authorization attempt.

## Effective permissions

To see the full set of permissions an agent has, including those received through delegation chains:

```typescript
const effective = await kavach.delegation.getEffectivePermissions(subAgent.id);
// Returns Permission[] combining the agent's own permissions and all active delegations
```

This is what the authorization engine uses internally. You can call it directly to understand what an agent can currently do.

## Listing chains

```typescript
// All active chains where subAgent is the receiver
const chains = await kavach.delegation.listChains({ toAgent: subAgent.id });

// All chains originating from an orchestrator
const outbound = await kavach.delegation.listChains({ fromAgent: orchestrator.id });
```

Each `DelegationChain` object includes:

```typescript
interface DelegationChain {
  id: string;
  fromAgent: string;
  toAgent: string;
  permissions: Permission[];
  expiresAt: Date;
  depth: number;      // current depth in the chain
  createdAt: Date;
}
```

## Typical pattern

An orchestrator agent receives a user request, plans a task, and delegates minimal permissions to short-lived sub-agents:

```typescript
// Orchestrator has broad permissions
const orchestrator = await kavach.agent.create({
  ownerId: 'user-123',
  name: 'planner',
  type: 'autonomous',
  permissions: [
    { resource: 'mcp:github:*', actions: ['read', 'write', 'comment'] },
    { resource: 'mcp:linear:*', actions: ['read', 'write'] },
  ],
});

// Sub-agent for a specific task — only gets what it needs
const codeReviewer = await kavach.agent.create({
  ownerId: 'user-123',
  name: 'code-reviewer',
  type: 'delegated',
  permissions: [],  // no direct permissions
});

await kavach.delegate({
  fromAgent: orchestrator.id,
  toAgent: codeReviewer.id,
  permissions: [
    { resource: 'mcp:github:pulls', actions: ['read', 'comment'] },
  ],
  expiresAt: new Date(Date.now() + 30 * 60_000),  // 30 minutes
  maxDepth: 1,  // cannot delegate further
});
```
